## Summary
Implemented a six digit OTP email code as the primary sign in path, entered in
the same tab, which fixes the Safari iOS failure. Magic links kept as a desktop
fallback. This supersedes the magic link only decision. Verified on a real
iPhone; broader mobile testing is still pending.

## What was done
- Added a code entry screen after the email step, same tab, no cross context link.
- Used `supabase.auth.signInWithOtp` in code mode; six digits, five minute expiry.
- Added rate limiting per email on the OTP request endpoint.
- Wired the OTP email template into Resend (see [[zone-email]]).
- Hid the magic link option on mobile user agents; kept it on desktop.
- Marked [[dec-0003]] superseded, set its `valid_to` to 2026-07-01.

## Decisions made
- OTP email code primary, magic link secondary. See [[dec-0004]], which
  supersedes [[dec-0003]].

## Scars
- None new. Reinforced the Safari iOS pattern captured in [[refl-safari-ios-auth]].

## Pending
- Test the OTP flow on a range of real mobile devices, not just one iPhone.
- Watch the Resend quota now that OTP shares it with receipts and invoice mail.

## Full log
- Considered native passwords as the fallback; rejected to stay passwordless and
  avoid reset flows.
- Root cause of the magic link failure is cross context link opening on Safari
  iOS; the fix is to never leave the originating tab, which OTP guarantees.
- This is the third Safari iOS incident; it is a recurring incompatibility, not
  three unrelated bugs. Flagged for the dream to reflect on.
